1. Цели и обоснование миграции
1.1 Что есть сейчас
Текущая версия системы построена на Google Apps Script. Это бесплатное и быстрое для прототипа решение, но имеет принципиальные ограничения:
Каждый проект — отдельная Google Sheet. При росте до 50+ проектов теряется управляемость
Apps Script имеет жёсткие лимиты квот: 6 минут на скрипт, 90 минут на пользователя в день, 50 МБ ответа
Нет реальной транзакционности — параллельная запись двух прорабов может затереть друг друга
Sheets не подходит для аналитики: нельзя индексировать, нельзя делать сложные join, медленные агрегации
Невозможна интеграция с 1С, банковскими API, складскими системами
Невозможно создание мобильного приложения для App Store / Google Play (нужен полноценный REST API)
Зависимость от показа баннеров Google в веб-приложениях («Это приложение создано пользователем Google Apps Script»)
Невозможен корпоративный single sign-on, OAuth для других сервисов компании
1.2 Что хотим получить

Бизнес-цели
Масштабируемая платформа на 100+ проектов и 200+ пользователей одновременно
Свой брендированный сайт и приложение на собственном домене welldom05.ru
Возможность продавать систему как SaaS для других строительных компаний
Интеграции с банками (платёжные документы), 1С, маркетплейсами стройматериалов
Аналитическая отчётность за любые периоды и проекты в реальном времени

Технические цели
Single source of truth — PostgreSQL как главное хранилище
Google Sheets сохраняется как опциональный экспорт по запросу администратора
Контейнеризация — деплой через Docker одной командой
CI/CD — автодеплой при push в main-ветку
Мониторинг и логи — Prometheus + Grafana, sentry для ошибок
Полная обратная совместимость с текущим PWA — клиент не заметит миграцию

2.2 Компоненты системы
Reverse Proxy (Caddy)
Caddy — современная альтернатива Nginx с автоматическим Let's Encrypt SSL. Принимает все входящие HTTPS-запросы, терминирует TLS, проксирует на FastAPI. Также раздаёт статику PWA (Index.html, манифест, иконки).
Application Server (FastAPI)
Главное приложение на Python 3.12. Запускается через uvicorn (ASGI-сервер) в нескольких воркерах для параллельной обработки запросов. Каждый воркер — отдельный процесс, общается с БД через пул соединений.
PostgreSQL 16
Основное хранилище. Используется primary + read replica для разделения нагрузки: запись идёт в primary, отчёты и аналитика читают с реплики. Бэкапы каждые 6 часов, WAL-архивирование для point-in-time recovery.
Redis
Используется для:
Сессии пользователей и JWT refresh-токены
Rate limiting (защита от подбора PIN, ограничение запросов на endpoint)
Кеширование часто запрашиваемых данных (справочники, агрегаты дашборда)
Pub/Sub для уведомлений в реальном времени (опционально)
S3-совместимое хранилище фото
Все фотографии чеков, квитанций и стройки хранятся в S3-совместимом объектном хранилище. Варианты:
Yandex Object Storage (рекомендуется для РФ)
Selectel S3, VK Cloud Object Storage
Self-hosted MinIO (для air-gap инсталляций)
Фото загружаются через presigned URL — клиент общается с S3 напрямую, минуя сервер. Это снимает нагрузку с FastAPI.
Google Sheets integration (опциональная)
Отдельный модуль `sheets_exporter`, который по запросу администратора формирует Google-таблицу проекта в формате, аналогичном текущей системе. Используется для:
Передачи отчётности клиенту в привычном для него виде
Резервного хранения данных в Google Drive
Бухгалтерской выгрузки (одна таблица на проект, привычный формат)
Sheets не является источником истины и используется только в режиме «read from PostgreSQL, write to Sheets».

3.4 Фронтенд
На первом этапе фронтенд остаётся текущим PWA — один HTML-файл с встроенными CSS и JavaScript. Меняется только endpoint API:

// БЫЛО (Apps Script):
google.script.run
  .withSuccessHandler(handleResult)
  .addExpense(data);
 
// СТАЛО (FastAPI):
fetch("https://api.welldom05.ru/api/v1/expenses", {
  method: "POST",
  headers: {
    "Authorization": "Bearer " + accessToken,
    "Content-Type": "application/json"
  },
  body: JSON.stringify(data)
}).then(r => r.json()).then(handleResult);
Для миграции создаётся универсальный wrapper `api.js`, который полностью повторяет интерфейс google.script.run — это позволяет минимизировать изменения в существующем коде PWA.

5. REST API endpoints
Все endpoints под префиксом /api/v1. Аутентификация через JWT-токены в заголовке Authorization: Bearer <token>. Версионирование через URL для безопасной эволюции API.
5.1 Аутентификация
Метод	Путь	Описание
POST	/api/v1/auth/login	Вход: phone + pin ? access_token, refresh_token
POST	/api/v1/auth/refresh	Обновление access_token по refresh_token
POST	/api/v1/auth/logout	Инвалидация refresh_token
POST	/api/v1/auth/change-pin	Смена своего PIN (требует старого)
GET	/api/v1/auth/me	Текущий пользователь + список проектов

7. Аутентификация и авторизация
7.1 Схема JWT
Используется пара токенов:
Access token — короткоживущий (15 минут), содержит user_id, role, permissions. Передаётся в каждом запросе.
Refresh token — долгоживущий (30 дней), хранится только на стороне клиента в HttpOnly cookie. Используется только для получения нового access token.
При логине Refresh-токен записывается в Redis с TTL — это позволяет принудительно разлогинивать пользователя (token revocation).
7.2 Хранение PIN
PIN-код прораба никогда не хранится в открытом виде. Используется bcrypt с work factor 12:

from passlib.context import CryptContext
 
pwd_ctx = CryptContext(schemes=["bcrypt"], deprecated="auto")
 
def hash_pin(pin: str) -> str:
    return pwd_ctx.hash(pin)
 
def verify_pin(pin: str, hash: str) -> bool:
    return pwd_ctx.verify(pin, hash)
 
# В БД:
# pin_hash = "$2b$12$KIXxPfnK6S7ZpoG9qV7HZuYjL..."
7.3 Защита от подбора (rate limiting)
При 3-х неверных попытках подряд — блокировка на 15 минут. Реализовано на двух уровнях:
Per-account: failed_attempts в БД, при 3 ? locked_until = now() + 15 min
Per-IP: счётчик в Redis с TTL 60 секунд, при >5 попыток/мин ? 429 Too Many Requests
7.4 RBAC (роли)
Роль	Что может	Чего не может
admin	Всё: пользователи, проекты, экспорт	—
foreman	CRUD записей в своих проектах, видит свой доход	Не видит других прорабов, чужие проекты
client	Read-only история своего проекта, фото, контакты	Не видит комиссию, рантье, мастеров

Существующие данные в Google Sheets необходимо перенести в PostgreSQL без потерь. Миграция проходит в одну сессию (downtime ~30 минут).
9.1 Этапы миграции

Этап 1: Подготовка (1 день)
Развёрнут production-сервер с PostgreSQL
Запущена FastAPI без публичного доступа
Применены все миграции схемы (alembic upgrade head

Резервный режим: на время миграции можно использовать бумажный блокнот

import gspread

from app.db import async_session
 
async def migrate_project(sheet_id: str, project_code: str):
    ss = gc.open_by_key(sheet_id)

10. Безопасность
10.1 Транспорт
Весь трафик через HTTPS (TLS 1.3, автоматический Let's Encrypt через Caddy)
HSTS заголовок с preload — браузер всегда подключается по HTTPS
HTTP redirected на HTTPS на уровне Caddy
10.2 Аутентификация
JWT с коротким TTL (15 минут access, 30 дней refresh)
Refresh tokens в Redis с возможностью отзыва
PIN хешируется bcrypt (work factor 12)
Rate limiting: 3 попытки ? блокировка аккаунта на 15 минут
Per-IP rate limit: 5 попыток входа/минута
10.3 Авторизация
RBAC через декораторы FastAPI (require_role)
Row-level security в PostgreSQL для критичных таблиц
Проверка project_id в каждом запросе — клиент не может получить чужой проект подменой ID
Маскирование PII (имена мастеров) на уровне сервиса перед отправкой клиенту
10.4 Защита от инъекций
SQL injection — невозможна, SQLAlchemy ORM всегда параметризует запросы
XSS — Pydantic валидирует все входящие данные, фронт делает escapeHTML на вывод
CSRF — JWT в Authorization header (не в cookie) исключает классический CSRF
Path traversal в загрузке фото — S3 keys генерируются сервером, клиент не контролирует имя файла
10.5 Защита данных
Бэкапы PostgreSQL шифруются (AES-256) перед загрузкой в S3
Service account для Google API в Docker secrets, не в репозитории
Секреты в `.env` исключены из Git, читаются из Docker secrets
PII (телефоны, имена) хранятся в БД, но логи не пишут PII в открытом виде
GDPR-совместимое удаление: API DELETE /users/{id} полностью стирает все данные с подтверждением
10.6 Мониторинг и реакция
Sentry для tracking ошибок в production (бесплатный план до 5k events/мес)
Slack/Telegram алерт при > 50 failed login за час (попытка брутфорса)
Алерт при > 100 5xx ошибок в час
Audit log пишет все CREATE/UPDATE/DELETE — можно расследовать любой инцидент

Итого: ~10 недель работы одного разработчика среднего уровня. С учётом отпусков и непредвиденных задач — 12–14 недель.
